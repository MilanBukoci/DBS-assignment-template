\section{Špecifikácia}
\label{sec:specifikacia}

Navrhovaný systém predstavuje internetový obchod (eshop) zameraný na predaj technických produktov, vrátane predaja opravených (repasovaných) zariadení a výkupu pokazených produktov na opravu.

% ── Subsections ──────────────────────────────────────────────────────────────
% Use \subsection{} and \subsubsection{} when you need more structure.

\subsection{Účel systému a riešený problém}

Systém rieši nasledovné problémy:

\begin{itemize}
    \item potrebu zákazníka vyhľadať a zakúpiť konkrétny produkt bez nutnosti fyzickej návštevy kamennej predajne,
    \item potrebu efektívneho porovnávania produktov (cena, stav, dostupnosť),
    \item evidenciu skladových zásob v reálnom čase,
    \item riadenie procesu opráv a opätovného predaja opravených produktov,
    \item správu komunikácie so zákazníkmi.
\end{itemize}

\subsection*{Cieľová skupina}

System je určený pre:

\begin{itemize}
    \item zákazníkov, ktorí preferujú online nákup techniky,
    \item zákazníkov hľadajúcich cenovo dostupnejšie (opravené) produkty,
    \item interných pracovníkov firmy zabezpečujúcich predaj, sklad, opravy a podporu.
\end{itemize}

\subsection*{Pridaná hodnota systému}

\begin{itemize}
    \item pohodlné vyhľadávanie a porovnávanie produktov,
    \item dostupnosť informácií o stave a skladovej zásobe v reálnom čase,
    \item možnosť nákupu lacnejších opravených produktov,
    \item efektívne riadenie opráv a priorít,
    \item centralizovaná evidencia objednávok, zákazníkov a komunikácie,
    \item zníženie administratívnej záťaže zamestnancov.
\end{itemize}

\subsubsection{Produkt}

Evidované údaje:

\begin{itemize}
    \item identifikátor produktu,
    \item názov,
    \item nákupná cena,
    \item predajná cena,
    \item stav (nový, pokazený, opravovaný, opravený),
    \item lokalita v sklade,
    \item dátum nákupu,
    \item počet kusov na sklade.
\end{itemize}

\subsection*{Dôležitosť}

Produkt je centrálna entita systému – predstavuje predmet predaja aj opravy.

\subsubsection{Zákazník}

Evidované údaje:

\begin{itemize}
    \item identifikátor zákazníka,
    \item meno a priezvisko,
    \item adresa,
    \item kontaktné údaje (napr. e-mail),
    \item dátum vytvorenia účtu.
\end{itemize}

\subsection*{Dôležitosť}

Umožňuje evidenciu objednávok, komunikácie a histórie nákupov.

\subsubsection{Objednávka}

Evidované údaje:

\begin{itemize}
    \item identifikátor objednávky,
    \item zákazník,
    \item dátum vytvorenia,
    \item adresa doručenia,
    \item stav objednávky (vytvorená, potvrdená, odoslaná, zrušená),
    \item zoznam položiek (produkt + množstvo).
\end{itemize}

\subsection*{Dôležitosť}

Zaznamenáva obchodnú transakciu medzi zákazníkom a systémom.

\subsubsection{Správa (technická podpora)}

Evidované údaje:

\begin{itemize}
    \item identifikátor správy,
    \item zákazník,
    \item dátum odoslania,
    \item obsah správy,
    \item stav vybavenia.
\end{itemize}

\subsection*{Dôležitosť}

Zabezpečuje evidenciu komunikácie a riešenie problémov.

\subsubsection{Oprava}

Evidované údaje:

\begin{itemize}
    \item identifikátor opravy,
    \item produkt,
    \item opravár,
    \item dátum prijatia na opravu,
    \item priorita (štandardná / zrýchlená),
    \item použitý materiál,
    \item dátum ukončenia,
    \item výsledný stav.
\end{itemize}

\subsection*{Dôležitosť}

Umožňuje riadiť proces opravy a spätné zaradenie produktu do predaja.

\subsection {Biznis pravidlá a obmedzenia}

\subsubsection{Pravidlá týkajúce sa predaja}

\begin{itemize}
    \item Nie je možné predať produkt so stavom \uv{pokazený} alebo \uv{opravovaný}.
    \item Nie je možné predať viac kusov produktu, než je aktuálne dostupné množstvo na sklade.
    \item Produkt môže byť zaradený do objednávky len ak je evidovaný v systéme.
\end{itemize}

\subsection*{Integritné obmedzenia}

\begin{itemize}
    \item kontrola stavu produktu pred vytvorením/potvrdením objednávky,
    \item kontrola dostupného množstva (množstvo $\geq 0$),
    \item referenčná integrita medzi objednávkou a produktom.
\end{itemize}

\subsubsection{Pravidlá týkajúce sa skladu}

\begin{itemize}
    \item Po potvrdení objednávky sa musí znížiť počet kusov na sklade.
    \item Po prijatí nového tovaru sa musí zvýšiť počet kusov na sklade.
    \item Stav produktu musí byť vždy jedna z povolených hodnôt (nový, pokazený, opravovaný, opravený).
\end{itemize}

\subsection*{Integritné obmedzenia}

\begin{itemize}
    \item množstvo na sklade nesmie byť záporné,
    \item obmedzenie domény pre atribút \uv{stav produktu}.
\end{itemize}

\subsubsection{Pravidlá týkajúce sa opráv}

\begin{itemize}
    \item Produkt môže byť zaradený do opravy len ak je v stave \uv{pokazený}.
    \item Po úspešnej oprave sa stav zmení na \uv{opravený}.
    \item Priorita opravy ovplyvňuje poradie v poradovníku.
\end{itemize}

\subsection*{Integritné obmedzenia}

\begin{itemize}
    \item väzba medzi entitou Oprava a existujúcim produktom,
    \item kontrola povolených prechodov stavov (stavový model).
\end{itemize}

\subsubsection{Pravidlá týkajúce sa prístupových práv}

\begin{itemize}
    \item Opravár nemá prístup k údajom o objednávkach.
    \item Zákazník má prístup len k vlastným údajom.
    \item Administrátor má plný prístup.
    \item Technická podpora nemôže meniť skladové množstvá ani finančné údaje.
\end{itemize}

\subsection*{Integritné obmedzenia}

\begin{itemize}
    \item implementácia rolí a oprávnení,
    \item kontrola prístupových práv na úrovni operácií nad entitami.
\end{itemize}

\section{Definovanie použivateľských scenárov}
\label{sec:definovanie_pouzivatelskych_scenarov}

\subsection{Vytvorenie objednávky zákazníkom}

\subsection*{Aktér}

Zákazník

\subsection*{Stručný popis}

Scenár popisuje proces, v ktorom zákazník vyberie produkty z ponuky eshopu a vytvorí objednávku. Zákazník môže pridať jeden alebo viac produktov do objednávky, zadať doručovaciu adresu a potvrdiť objednávku. Po potvrdení objednávky systém uloží objednávku do databázy a aktualizuje množstvo produktov na sklade.

\subsection*{Hlavný tok scenára}

\begin{enumerate}
    \item Zákazník si prehliada ponuku produktov dostupných v eshope.
    \item Zákazník vyberie jeden alebo viac produktov, ktoré chce zakúpiť.
    \item Zákazník zadá požadované množstvo pre každý produkt.
    \item Zákazník prejde do procesu vytvorenia objednávky.
    \item Zákazník zadá doručovaciu adresu.
    \item Systém overí dostupnosť produktov na sklade.
    \item Zákazník potvrdí vytvorenie objednávky.
    \item Systém vytvorí novú objednávku v databáze.
    \item Systém vytvorí položky objednávky pre jednotlivé produkty.
    \item Systém aktualizuje množstvo produktov na sklade.
    \item Objednávka je uložená v systéme so stavom \uv{vytvorená}.
\end{enumerate}

\subsection*{Čo sa deje na dátovej vrstve}

Počas realizácie scenára prebiehajú nasledujúce operácie nad databázou:

\begin{itemize}
    \item vyhľadanie zákazníka v tabuľke \textbf{Zákazník}
    \item vyhľadanie produktov v tabuľke \textbf{Produkt}
    \item kontrola dostupného množstva produktu (\textbf{Produkt.mnozstvo})
    \item vytvorenie nového záznamu v tabuľke \textbf{Objednávka}
    \item vytvorenie jedného alebo viacerých záznamov v tabuľke \textbf{Objednavka\_Produkt}
    \item aktualizácia počtu kusov produktu v tabuľke \textbf{Produkt} (zníženie hodnoty \textit{mnozstvo})
\end{itemize}

\subsection*{Alternatívne toky}

\textbf{A1 - Nedostatočné množstvo na sklade}

\begin{enumerate}
    \item (2.1) Systém pri kontrole dostupnosti zistí, že požadované množstvo produktu nie je dostupné.
    \item (2.2) Systém informuje zákazníka o nedostupnosti produktu.
    \item (2.3) Zákazník môže:
    \begin{itemize}
        \item znížiť požadované množstvo produktu,
        \item odstrániť produkt z objednávky,
        \item alebo zrušiť vytváranie objednávky.
    \end{itemize}
\end{enumerate}

\noindent \textbf{A2 - Neplatná doručovacia adresa}

\begin{enumerate}
    \item (5.1) Zákazník zadá neúplnú alebo neplatnú adresu.
    \item (5.2) Systém zobrazí upozornenie a vyzve zákazníka na opravu údajov.
    \item (5.3) Proces vytvorenia objednávky pokračuje po opravení údajov.
\end{enumerate}

Pre tento scenár je najvhodnejší \textbf{sekvenčný diagram (Sequence Diagram)}, ako je ukázané na obrázku~\ref{fig:objednavka}.

\noindent Sekvenčný diagram umožňuje znázorniť:

    \item interakciu medzi zákazníkom a systémom,
    \item postupnosť operácií pri vytváraní objednávky,
    \item komunikáciu medzi jednotlivými komponentmi systému a databázou.
\end{itemize}

\noindent Diagram znázorňuje najmä:

\begin{itemize}

\end{itemize}

\subsection{Pridelenie pokazeného produktu na opravu}

\subsection*{Aktér}

Administrátor

\subsection*{Stručný popis}

Scenár popisuje proces, v ktorom administrátor zaradí pokazený produkt do procesu opravy. Administrátor vyberie pokazený produkt zo systému a priradí ho konkrétnemu opravárovi. Následne systém vytvorí záznam o oprave, ktorý bude opravár spracovávať.

\subsection*{Hlavný tok scenára}

\begin{enumerate}
    \item Administrátor sa prihlási do systému.
    \item Administrátor zobrazí zoznam produktov evidovaných v systéme.
    \item Administrátor vyfiltruje produkty so stavom \uv{pokazený}.
    \item Administrátor vyberie produkt, ktorý má byť opravený.
    \item Administrátor vyberie opravára, ktorý bude za opravu zodpovedný.
    \item Administrátor nastaví prioritu opravy.
    \item Systém vytvorí nový záznam o oprave.
    \item Systém priradí zvoleného opravára k oprave.
    \item Oprava je uložená v systéme a zaradená do poradovníka opráv.
\end{enumerate}

\subsection*{Čo sa deje na dátovej vrstve}

Pri realizácii scenára systém vykonáva nasledujúce operácie nad databázou:

\begin{itemize}
    \item vyhľadanie produktu v tabuľke \textbf{Produkt}
    \item kontrola stavu produktu \textbf{(status = pokazený)}
    \item vyhľadanie opravára v tabuľke \textbf{Opravár}
    \item vytvorenie nového záznamu v tabuľke \textbf{Oprava}
    \item uloženie identifikátora produktu (\textbf{id\_tovaru}) a opravára (\textbf{id\_opravara}) do záznamu opravy
\end{itemize}

\subsection*{Alternatívne toky}

\textbf{A1 – Produkt nie je v stave pokazený}

\begin{enumerate}
    \item (4.1) Systém zistí, že stav produktu nie je \uv{pokazený}.
    \item (4.2) Systém zobrazí upozornenie.
    \item (4.3) Administrátor musí vybrať iný produkt.
\end{enumerate}

\noindent \textbf{A2 - Opravár nie je dostupný}

\begin{enumerate}
    \item (5.1) Zvolený opravár má už maximálny počet pridelených opráv.
    \item (5.2) Administrátor musí vybrať iného opravára.
\end{enumerate}

\subsection*{Podporný UML diagram}

Pre tento scenár je vhodný \textbf{diagram aktivít (Activity Diagram)}, ako je ukázané na obrázku~\ref{fig:oprava}.

\noindent Diagram aktivít umožňuje znázorniť:

\begin{itemize}
    \item postup jednotlivých krokov procesu,
    \item rozhodovacie body (napr. kontrola stavu produktu),
    \item tok procesu od výberu produktu až po vytvorenie opravy.
\end{itemize}

\noindent Diagram jasne znázorňuje logiku procesu priraďovania opravy a rozhodovacie podmienky v systéme.


\subsection{Oprava tovaru}
\subsection*{Aktér}
Opravár

\subsection*{Stručný popis}
Opravár si vyberie produkt na opravu, ktorý je priradený konkrétne jemu. Potom zistí, ktoré náhradné diely potrebuje a zoberie si ich zo skladu. Následne tovar opraví a vráti ho do skladu. Údaje o konkrétnom tovare budú po oprave aktualizované a uložené do súvisiacich tabuliek.

\subsection*{Vstupné podmienky}
\begin{itemize}
    \item Opravár je prihlásený do systému
    \item V tabuľke opráv sú nedokončené opravy (opravy bez dátumu ukončenia), ktoré patria prihlásenému technikovi
\end{itemize}

\subsection*{Výstupné podmienky}
\begin{itemize}
    \item Konkrétny tovar je opravený, naskladnený a záznamy súvisiace s ním sú aktualizované
    \item Záznam o oprave je aktualizovaný a má vyplnený dátum ukončenia
    \item Množstvo použitých náhradných dielov je zaznamenané a súvisiace záznamy sú aktualizované
\end{itemize}

\subsection*{Hlavný tok scenára}
\begin{enumerate}
    \item Opravár otvorí v systéme zoznam svojich opráv
    \item Systém zobrazí dostupné možnosti (najskôr so zrýchlenou prioritou)
    \item Opravár si vyberie konkrétnu opravu
    \item Opravár vykoná diagnostiku zariadenia a zistí, aké náhradné diely potrebuje
    \item Systém zobrazí všetky dostupné diely na sklade a údaje o nich
    \item Opravár skontroluje, či sú potrebné diely na sklade, a zarezervuje si ich
    \item Opravár si fyzicky zoberie diely a vykoná opravu
    \item Po skončení opravy, opravár zadá dátum ukončenia a označí ju ako hotovú
    \item Systém zmení stav tovaru na \uv{Naskladnený} a zmení jeho \textbf{id\_produktu} na rovnaký model, ale so statusom \uv{Opravený}
\end{enumerate}

\subsection*{Čo sa deje na dátovej vrstve:}
\begin{enumerate}
    \item Načítanie záznamov z tabuľky Oprava, ktoré nemajú vyplnený \textbf{datum\_ukoncenia} a v stĺpci \textbf{id\_opravara} majú ID aktuálne prihláseného technika
    \item Načítanie jedného konkrétneho záznamu z tabuľky Oprava
    \item Systém načíta všetky záznamy z tabuľky \textbf{Nahradny\_Diel}
    \item Pridanie nových záznamov do prepojovacej tabuľky \textbf{Oprava\_Diel}
    \item Aktualizácia atribútu množstvo v tabuľke \textbf{Nahradny\_Diel}
    \item Aktualizácia atribútu \textbf{datum\_ukoncenia} v zázname tejto opravy v tabuľke Oprava

\section{Návrh relačného modelu}
\label{sec:navrh_relacneho_modelu}

Diagram relačného modelu našej databázy~\ref{fig:physical_model} pozostáva z 9 základných tabuliek a dvoch prepojovacích tabuliek (Objednavka\_Tovar, Oprava\_Diel).
V každej z tabuliek sú uvedené všetky potrebné atribúty. Not Null sú označené hviezdičkami (*) (krúžky na diagrame) a všetky ostatné obmedzenia a kľúče sa nachádzajú pri typoch údajov v lomených zátvorkách (napr. <<PK>>, <<FK>>, <<CK>>).



\subsection{Detailný popis tabuliek}

\subsubsection{Zákazník}
\textbf{Účel v kontexte systému:}
Tabuľka obsahuje záznamy všetkých používateľov systému. Slúži na ukladanie ich údajov, ktoré sú neskôr potrebné na spracovanie objednávky alebo na zaslanie správy podpore.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id : UUID  <<PK>>} - \textbf{Not Null}, Primary Key
    \item \texttt{email : VARCHAR(254) <<UQ>>} - \textbf{Not Null}, Unique (každý zákazník musí mať unikátny email, na jeden môže byť zaregistrovaný iba jeden účet)
    \item \texttt{tel\_cislo : VARCHAR(15) <<UQ>>} - Unique (takisto ako email, musí byť unikátny, ale nie je povinný)
    \item \texttt{meno : VARCHAR(100)} - (nie je povinný údaj, zákazník si ho môže uviesť už v objednávke alebo neskôr)
    \item \texttt{adresa : VARCHAR(255)} - (tiež je len pre pohodlie zákazníka, ak ho zadá - bude automaticky zadávaný do objednávky)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Každý zákazník dostane pri registrácii unikatné id (umelý kľúč). To je potrebné na to, aby mohol zákazník meniť všetky ostatné údaje a účet zostal rovnaký.

\noindent \textbf{Cudzie kľúče:}
Táto tabuľka nemá cudzie kľúče.



\subsubsection{Pracovník}
\textbf{Účel v kontexte systému:}
Tabuľka slúži na evidenciu zamestnancov. Obsahuje všetky pracovné pozície (podpora, opravári, administrátori) v jednej tabuľke.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id : UUID <<PK>>} - \textbf{Not Null}, Primary Key
    \item \texttt{meno : VARCHAR(100)} - \textbf{Not Null} (meno zamestnanca)
    \item \texttt{pozicia : ENUM('Podpora', 'Opravar', 'Administrator')} - \textbf{Not Null} (určuje rolu zamestnanca v systéme)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Každý pracovník dostane unikátne id vo formáte UUID (umelý kľúč). Nie je možné použiť napr. meno, pretože sa môže stať, že niekoľko zamestancov bude mať rovnaké meno.

\noindent \textbf{Cudzie kľúče:}
Táto tabuľka nemá cudzie kľúče



\subsubsection{Správa}
\textbf{Účel v kontexte systému:}
Tabuľka slúži na ukladanie správ a požiadaviek od zákazníkov pre technickú podporu. Umožňuje začať komunikáciu a sledovať stav riešenia problémov.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id : UUID  <<PK>>} - \textbf{Not Null}, Primary Key
    \item \texttt{id\_zakaznika : UUID <<FK>>} - \textbf{Not Null}, Foreign Key (prepája správu s konkrétnym zákazníkom, ktorý ju vytvoril)
    \item \texttt{id\_pracovnika : UUID <<FK>>} - Foreign Key (ukazuje na pracovníka podpory, ktorý potom bude túto správu riešiť)
    \item \texttt{datum\_vytvorenia : TIMESTAMP <<DF>>} - \textbf{Not Null}, Default (automaticky zaznamená presný čas vytvorenia správy systémom)
    \item \texttt{sprava : TEXT} - \textbf{Not Null} (samotný text požiadavky alebo problému od zákazníka)
    \item \texttt{status : ENUM('Nova', 'Dokoncena') <<DF>> DEFAULT 'Nova'} - \textbf{Not Null}, Default (každá nová správa automaticky má stav 'Nova', pokiaľ ju podpora nevyrieši)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Každá správa dostane pri vytvorení unikátne id (umelý kľúč). Slúži na presnú identifikáciu konkrétnej správy v systéme, aj keby zákazník poslal niekoľko správ s identickým textom v rovnaký čas.

\noindent \textbf{Cudzie kľúče:}
Tabuľka obsahuje dva cudzie kľúče: \texttt{id\_zakaznika}, ktorý odkazuje na primárny kľúč v tabuľke \texttt{Zakaznik -> id} a \texttt{id\_pracovnika}, ktorý ukazuje na primárny kľúč v tabuľke \texttt{Pracovnik -> id}. Prvý kľúč identifikuje zákazníka, ktorý správu odoslal, a druhý určuje pracovníka podpory, ktorý na ňu odpovedal (respektíve aj upravoval objednávku).



\subsubsection{Objednávka}
\textbf{Účel v kontexte systému:}
Tabuľka slúži na uchovávanie všetkých objednávok, ktoré zákazníci vytvorili. Ukladá informácie o doručení, čase nákupu a aktuálnom stave objednávky.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id : UUID  <<PK>>} - \textbf{Not Null}, Primary Key
    \item \texttt{id\_zakaznika : UUID <<FK>>} - \textbf{Not Null}, Foreign Key (prepája objednávku s konkrétnym zákazníkom)
    \item \texttt{datum\_vytvorenia : TIMESTAMP <<DF>>} - \textbf{Not Null}, Default (automaticky zaznamená presný čas vytvorenia objednávky)
    \item \texttt{adresa\_dorucenia : VARCHAR(255)} - \textbf{Not Null} (fyzická adresa, na ktorú má byť tovar doručený)
    \item \texttt{status : ENUM('Vytvorena', 'Zrusena', 'Dokoncena') <<DF>> DEFAULT 'Vytvorena'} - \textbf{Not Null}, Default (každá nová objednávka automaticky dostáva stav 'Vytvorena', pokiaľ nebude spracovaná, alebo zákazník ju nezruší)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Každá objednávka dostane pri vytvorení unikátne id (umelý kľúč). Slúži na identifikáciu nákupu a umožňuje bezpečne spájať s tovarami cez prepojovaciu tabuľku bez rizika duplicity.

\noindent \textbf{Cudzie kľúče:}
Tabuľka obsahuje jeden cudzí kľúč: \texttt{id\_zakaznika}, ktorý odkazuje na primárny kľúč v tabuľke \texttt{Zakaznik -> id}. Ukazuje na konkrétneho existujúceho zákazníka, ktorý túto objednávku vytvoril.



\subsubsection{Tovar}
\textbf{Účel v kontexte systému:}
Tabuľka slúži na ukladanie fyzických kusov produktu, ktoré máme na sklade. Pomáha sledovať stav každého z nich a konkrétne miesto jeho naskladnenia.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id : UUID  <<PK>>} - \textbf{Not Null}, Primary Key
    \item \texttt{id\_produktu : UUID <<FK>>} - \textbf{Not Null}, Foreign Key (ukazuje, o aký typ produktu ide, napr. Iphone 17 Pro Max opravený/nový atď)
    \item \texttt{status : ENUM('Naskladneny', 'Predany', 'Pokazeny', 'Opravovany')} - \textbf{Not Null} (stav tohto kusu)
    \item \texttt{miesto\_naskladnenia :VARCHAR(255)} - \textbf{Not Null} (miesto v našom sklade, napr. regál a konkrétneho miesto na ňom, aby ho zamestnanci vedeli nájsť)
    \item \texttt{datum\_nakupu : TIMESTAMP <<DF>>} - Default (timestamp kedy tento kus bol prijaty na sklad)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Každý fyzický kus tovaru dostane vlastné unikátne id (umelý kľúč). Funguje to ako čiarový kód. Je to potrebné, pretože môžeme mať na sklade 10 rovnakých napr. mobilov, ale každý potrebujeme evidovať zvlášť (kvôli tomu, že môžu mať rôzny stav a môžu byť viackrát opravované).

\noindent \textbf{Cudzie kľúče:}
Tabuľka obsahuje jeden cudzí kľúč: \texttt{id\_produktu}, ktorý odkazuje na primárny kľúč v tabuľke \texttt{Produkt -> id}. Ukazuje, ku ktorému typu produktu (modelu v katalógu) tento kus patrí (napr. Google Pixel 10a opravený, alebo Google Pixel 10a nový).



\subsubsection{Produkt}
\textbf{Účel v kontexte systému:}
Tabuľka slúži ako katalóg produktov v našom e-shope. Nie je pre fyzické kusy, ale len ich typ (model), ktorý máme v predaji. Tieto záznamy sa využívajú napr. na webe nášho e-shopu pri zobrazovaní produktu.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id : UUID  <<PK>>} - \textbf{Not Null}, Primary Key
    \item \texttt{nazov : VARCHAR(255)} - \textbf{Not Null} (názov modelu, ktorý sa zobrazí na stránke)
    \item \texttt{status : ENUM('Novy', 'Opraveny')} - \textbf{Not Null} (rozdeľuje produkty v katalógu na úplne nové a tie, ktoré predávame po oprave (zánovné))
    \item \texttt{mnozstvo : INT <<CK>> CHECK(mnozstvo >= 0)} - \textbf{Not Null}, Check (aktuálny počet dostupných kusov tohto modelu na sklade. Ochrana databázy nedovolí znížiť množstvo tovaru až pod nulu aby nebolo možné predať tovar, ktorý nemáme)
    \item \texttt{cena : DECIMAL(8, 2)} - \textbf{Not Null} (aktuálna cena, tiež bude zobrazená na webe)
    \item \texttt{cena\_pri\_nakupe : DECIMAL(8, 2)} - (cena len pre účtovníkov, aby sme vedeli vypočítať koľko sme zarobili po predávaní kusu tovaru, nie je povinná)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Každý produkt v katalógu dostane unikátne id (umelý kľúč). Je to lepšie ako použiť samotný názov produktu, pretože názov sa môže zmeniť. Taktiež môže byť produkt nový alebo opravený, čo nechceme zobrazovať v názve a bez tohto stavu by boli uložené v systéme ako identické aj ak nie sú

\noindent \textbf{Cudzie kľúče:}
Táto tabuľka nemá žiadne cudzie kľúče.



\subsubsection{Oprava}
\textbf{Účel v kontexte systému:}
Tabuľka slúži na zaznamenanie procesu opravy. Ak dostaneme napríklad pokazený mobil, tu presne zaznamenáme, kto ho má opraviť, ako rýchlo a kedy oprava bola začatá.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id : UUID  <<PK>>} - \textbf{Not Null}, Primary Key
    \item \texttt{id\_tovaru : UUID <<FK>>} - \textbf{Not Null}, Foreign Key (ukazuje na fyzický kus tovaru, ktorý leží u opravara na stole)
    \item \texttt{id\_opravara : UUID <<FK>>} - \textbf{Not Null}, Foreign Key (ukazuje na zamestnanca, ktorý túto opravu dostal)
    \item \texttt{priorita : ENUM('Standardna', 'Zrychlena')} - \textbf{Not Null} (ukazuje na to, či musí byť tovar opravený čo najrýchlejšie alebo má čakať v rade na opravu)
    \item \texttt{datum\_prijatia : TIMESTAMP <<DF>>} - \textbf{Not Null}, Default (automaticky zaznamená presný čas kedy bolo zariadenie prijaté na opravu)
    \item \texttt{datum\_ukoncenia : DATE} - (predpokladaný čas ukončenia opravy alebo aj dátum, kedy bola oprava reálne ukončená, nemusí byť zadaný od začiatku)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Každá oprava dostane unikátne id (umelý kľúč). Je to nutné, pretože jedno zariadenie môže byť opravované viackrát a musíme zaznamenať každu takúto opravu.

\noindent \textbf{Cudzie kľúče:}
Tabuľka obsahuje dva cudzie kľúče. \texttt{id\_tovaru} odkazuje na primárny kľúč v tabuľke \texttt{Tovar -> id} (konkrétny pokazený kus). \texttt{id\_opravara} odkazuje na \texttt{Pracovnik -> id} (konkrétny opravár, ktorý to rieši).



\subsubsection{Náhradný diel}
\textbf{Účel v kontexte systému:}
Tabuľka funguje ako sklad súčiastok. Keď opravár potrebuje napríklad novú batériu alebo displej pre pokazený mobil, systém skontroluje, či ju máme, koľko ich ešte zostalo a kde presne je naskladnena.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id : UUID  <<PK>>} - \textbf{Not Null}, Primary Key
    \item \texttt{nazov : VARCHAR(255)} - \textbf{Not Null} (ako sa súčiastka presne nazýva, napr. \uv{Displej: Xiaomi 15T Pro Max Ultra 3D})
    \item \texttt{mnozstvo : INT} - \textbf{Not Null} (koľko kusov tejto súčiastky aktuálne máme na sklade)
    \item \texttt{cena : DECIMAL(8, 2)} - \textbf{Not Null} (koľko táto súčiastka stála, je to dôležité pre výpočet ceny celkovej opravy)
    \item \texttt{miesto\_naskladnenia : VARCHAR(255)} - \textbf{Not Null} (presná poloha na sklade, napr. miesto na regále, aby ju opravár rýchlo našiel)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Každý typ diela dostane svoje unikátne id (umelý kľúč). Ich názvy môžu byť dlhé alebo podobné, takže unikátny identifikátor je najbezpečnejší a najrýchlejší spôsob, ako v systéme presne identifikovať konkrétny diel.

\noindent \textbf{Cudzie kľúče:}
Táto tabuľka nemá žiadne cudzie kľúče.



\subsubsection{Poziadavka\_Na\_Diely}
\textbf{Účel v kontexte systému:}
Tabuľka slúži na evidenciu požiadaviek. Ak opravár v procese opravy zisty, že mu chyba nejaký náhradný diel, tak výtvory požiadavku, ktorú administrátor už následne spracuje a objedná potrebné náhradné diely.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id : UUID <<PK>>} - \textbf{Not Null}, Primary Key
    \item \texttt{id\_diel : UUID <<FK>>} - \textbf{Not Null}, Foreign Key (ukazuje, aký konkrétny diel chýba)
    \item \texttt{id\_opravy : UUID <<FK>>} - \textbf{Not Null}, Foreign Key (ukazuje, pre ktorú opravu je tento diel potrebný)
    \item \texttt{mnozstvo : INT <<CK>> CHECK(mnozstvo > 0)} - \textbf{Not Null}, Check (počet požadovaných kusov, nesmie byť nula alebo záporné číslo)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Používa sa unikátne id vo formáte UUID. Taký výber je kvôli potrebe mať tieto požiadavky vždy unikátne, pretože niekedy môžu vzniknúť niekoľko podobných/takých istých požiadaviek.

\noindent \textbf{Cudzie kľúče:}
Tabuľka obsahuje dva cudzie kľúče. \texttt{id\_diel} odkazuje na \texttt{Nahradny\_Diel -> id}. Kľúč \texttt{id\_opravy} odkazuje na \texttt{Oprava -> id}.



\subsubsection{Objednávka\_Tovar}
\textbf{Účel v kontexte systému:}
Táto prepojovacia tabuľka spája objednávky s konkrétnymi kusmi tovaru zo skladu. Pomáha zistiť, ktoré konkrétne zariadenia (napríklad mobil s konkrétnym ID) boli priradené k tejto objednávke.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id\_objednavky : UUID <<PK, FK>>} - \textbf{Not Null}, Primary Key, Foreign Key (ukazuje, ku ktorej objednávky tento tovar patrí)
    \item \texttt{id\_tovaru : UUID <<PK, FK>>} - \textbf{Not Null}, Primary Key, Foreign Key (ukazuje na konkrétny kus tovaru, ktorý zašleme v rámci tejto objednávky)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
Používame zložený primárny kľúč (kombinácia \texttt{id\_objednavky} a \texttt{id\_tovaru}). Zabraňuje tomu, aby sme jednej objednávke nepriradili ten istý kus tovaru dvakrát. Fyzicky totiž existuje len jeden kus tovaru a jeho viacnásobné priradenie k tej istej objednávke by spôsobilo chybu.

\noindent \textbf{Cudzie kľúče:}
Tabuľka obsahuje dva cudzie kľúče. \texttt{id\_objednavky} odkazuje na tabuľku \texttt{Objednavka -> id}. Druhý kľúč \texttt{id\_tovaru} odkazuje na tabuľku \texttt{Tovar -> id}.



\subsubsection{Oprava\_Diel}
\textbf{Účel v kontexte systému:}
Táto prepojovacia tabuľka slúži ako záznam o tom, koľko náhradných dielov sa použilo na konkrétnu opravu. Prepája opravu so skladom náhradných dielov. Zaznamenáva, aké súčiastky a v akom počte opravár použil na opravu zariadenia.

\noindent \textbf{Atribúty a obmedzenia:}
\begin{itemize}
    \item \texttt{id\_opravy : UUID <<PK, FK>>} - \textbf{Not Null}, Primary Key, Foreign Key (ukazuje, o ktorú opravu ide)
    \item \texttt{id\_nahradneho\_diela : UUID <<PK, FK>>} - \textbf{Not Null}, Primary Key, Foreign Key (ukazuje, ktorá konkrétna súčiastka bola použitá a už na sklade sa nenachádza)
    \item \texttt{mnozstvo : INT <<CK>> CHECK(mnozstvo > 0)} - \textbf{Not Null}, Check (koľko kusov tejto súčiastky bolo použito, napr. 2 rovnaké skrutky. Databáza je chránená a nedovolí zadať nulu alebo záporné číslo)
\end{itemize}
 
\noindent \textbf{Voľba primárneho kľúča:}
V tejto tabuľke nemáme jedno umelé id, ale používame zložený primárny kľúč (kombinácia \texttt{id\_opravy} a \texttt{id\_nahradneho\_diela}). Zabezpečuje to, že ten istý náhradný diel nemôžeme do jednej opravy zapísať omylom dvakrát ako dva samostatné záznamy. Ak opravár použije viac kusov rovnakého dielu, iba sa zvýši hodnota v atribúte množstvo.

\noindent \textbf{Cudzie kľúče:}
Tabuľka obsahuje dva cudzie kľúče. \texttt{id\_opravy} odkazuje na tabuľku \texttt{Oprava -> id}. Druhý kľúč \texttt{id\_nahradneho\_diela} odkazuje na tabuľku \texttt{Nahradny\_Diel -> id}.



\subsection{Popis vzťahov medzi tabuľkami}

V našom relačnom modeli sa nachádzajú primárne vzťahy typu \textbf{1:N} a \textbf{M:N} (pre nich boli vytvorené prepojovacie tabuľky). Vzťahy typu \textbf{1:1} sa v návrhu nenachádzajú, keďže neboli pre našu databázu potrebné (aj tak väčšinou sa dajú spojiť do jednej tabuľky)

\subsubsection{Vzťahy typu 1:N (Jeden k viacerým)}
Tento typ vzťahu je v databáze najčastejší. Realizuje sa na fyzickej úrovni tak, že do tabuľky na strane \uv{N} (viac) sa pridá cudzí kľúč (\textbf{FK}), ktorý odkazuje na primárny kľúč (\textbf{PK}) v tabuľke na strane \uv{1}.

\noindent \textbf{Príklady a kardinalita:}
\begin{itemize}
    \item \textbf{Zákazník (1) - (0..N) Objednávka:} Jeden zákazník môže mať nula až nekonečno objednávok. Každá objednávka však musí patriť práve jednému zákazníkovi.
    \item \textbf{Produkt (1) - (0..N) Tovar:} Jeden typ produktu v katalógu môže mať na sklade viacero fyzických kusov. Fyzický kus tovaru musí patriť jednému konkretnemu produktu.
    \item \textbf{Pracovník (1) - (0..N) Oprava:} Jeden pracovník môže byť zodpovedný za viacero opráv, ale konkrétna oprava je priradená iba jednému pracovníkovi.
\end{itemize}

\noindent \textbf{Správanie pri mazaní (RESTRICT):}
Pri všetkých vzťahoch 1:N v našom systéme využívame pravidlo \textbf{RESTRICT}. Znamená to, že databáza nedovolí vymazať napríklad Zákazníka, ak už má v systéme vytvorenú Objednávku alebo Správu. Je to kritické pre zachovanie finančnej a historickej konzistencie (nemôžeme stratiť históriu opráv len preto, že sa vymazal pracovník).

\subsubsection{Vzťahy typu M:N (Viacerí k viacerým)}
Tento typ vzťahu nie je možné v relačnej databáze realizovať priamo. Na fyzickej úrovni sa realizuje vytvorením \textbf{prepojovacej tabuľky}, ktorá tento vzťah rozbije na dva vzťahy 1:N.

\noindent \textbf{Príklady a realizácia:}
\begin{itemize}
    \item \textbf{Objednávka (M) - (N) Tovar:} Jedna objednávka môže obsahovať viacero kusov tovaru a teoreticky jeden kus tovaru (ak by bola objednávka zrušená a tovar vrátený, alebo ak jeden tovar potom opravíme a znovy predáme) môže vznikať v histórii viacerých objednávok. Realizované cez prepojovaciu tabuľku \texttt{Objednavka\_Tovar}.
    \item \textbf{Oprava (M) - (N) Náhradný diel:} Pri jednej oprave sa môže použiť viacero rôznych dielov a ten istý typ dielu sa používa pri viacerých opravách. Realizované cez prepojovaciu tabuľku \texttt{Oprava\_Diel}.
\end{itemize}

\noindent \textbf{Správanie pri mazaní (CASCADE):}
Pri prepojovacích tabuľkách využívame pravidlo \textbf{CASCADE}. Ak sa napríklad z nejakého dôvodu zmaže celá \texttt{Oprava}, databáza automaticky vymaže aj súvisiace záznamy v prepojovacej tabuľke \texttt{Oprava\_Diel}. Tým zabránime vzniku zaznamov, ktoré nikam nepatria, ktoré by ukazovali na už neexistujúcu opravu.



\subsection{Zabezpečenie integritných obmedzení}
Integritné obmedzenia z predchádzajúcich kapitol sú zabezpečené priamo na fyzickej vrstve databázy pomocou nasledujúcich mechanizmov:

\begin{itemize}
    \item \textbf{Entitná integrita:} Každá tabuľka má definovaný primárny kľúč \textbf{PK} vo formáte UUID, čo zaručuje absolútnu jedinečnosť každého záznamu v systéme.
    \item \textbf{Referenčná integrita:} Je zabezpečená pomocou cudzích kľúčov \textbf{FK}. Databáza nedovolí vytvoriť záznam odkazujúci na neexistujúce dáta a pravidlá \textbf{RESTRICT} a \textbf{CASCADE} chránia pred stratou prepojení pri mazaní.
    \item \textbf{Doménová integrita:} Využitie dátových typov \textbf{ENUM} obmedzuje hodnoty atribútov (napríklad status objednávky alebo pozícia pracovníka) len na vopred definovaný a povolený zoznam hodnôt.
    \item \textbf{Obmedzenia hodnôt:} Použitie klauzuly \textbf{CHECK} zaručuje logickú správnosť dát (napríklad množstvo na sklade nesmie klesnúť pod nulu), \textbf{UNIQUE} zabraňuje duplicitám pri emailoch a \textbf{NOT NULL} pre vyplnenie povinných poli.
\end{itemize}

\subsection{Realizácia používateľských scenárov}
Áno, navrhnutý relačný model plne pokrýva všetky identifikované používateľské príbehy a prípady použitia. Tabuľky a ich vzťahy boli modelované priamo na základe požiadaviek systému:

\begin{itemize}
    \item \textbf{Vytvorenie objednávky zákazníkom:} Model umožňuje prepojiť zákazníka s novou objednávkou, priradiť k nej konkrétne kusy tovaru cez prepojovaciu tabuľku a aktualizovať stav produktu na sklade.
    \item \textbf{Pridelenie pokazeného produktu na opravu:} Databáza umožňuje vyfiltrovať pokazený tovar, vytvoriť nový záznam o oprave a priradiť k nemu konkrétneho zamestnanca (opravára).
    \item \textbf{Oprava tovaru:} Systém eviduje spotrebu náhradných dielov pre konkrétnu opravu cez prepojovaciu tabuľku, umožňuje zmeniť stav tovaru po oprave a vytvoriť požiadavku na chýbajúce diely.
    \item \textbf{Komunikácia zákazníka s technickou podporou:} Model priamo spája zákazníka, jeho správy a pracovníka podpory, pričom pracovníkovi umožňuje prístup k histórii objednávok na prípadnú úpravu údajov.
\end{itemize}

\section{Zhodnotenie}
\label{sec:zhodnotenie}

Najviac času nám zabrala tvorba samotného relačného modelu databázy. Pri jeho návrhu bolo potrebné správne identifikovať entity, ich atribúty a najmä vzťahy medzi nimi tak, aby model správne reprezentoval fungovanie navrhovaného systému. Zároveň bolo potrebné dbať na logickú štruktúru modelu a vyhnúť sa zbytočnej redundancii dát.

Na začiatku práce sme mali najväčší problém s presným zameraním nášho e-shopu a s celkovým konceptom riešenia. Nebolo úplne jasné, aké funkcionality má systém pokrývať a aké entity budú potrebné. Tento problém sme vyriešili postupným spresňovaním špecifikácie systému a diskusiou o jednotlivých scenároch používania, čo nám pomohlo lepšie pochopiť riešeniu.

Pre nás bola úplne novou skúsenosťou tvorba relačného modelu databázy. Pri jeho návrhu sme sa museli naučiť správne identifikovať primárne a cudzie kľúče, ako aj vhodne modelovať vzťahy medzi tabuľkami.

Navrhnuté riešenie podľa nášho názoru pokrýva zvolenú tému v dostatočnej miere. Počas návrhu sme však niektoré časti zjednodušili. Na začiatku sme napríklad uvažovali o samostatnej tabuľke pre každého aktéra systému (administrátor, opravár a technická podpora). Neskôr sme sa rozhodli tento návrh zjednodušiť a vytvoriť jednu spoločnú tabuľku pracovník, v ktorej je každému pracovníkovi priradená konkrétna rola.

Náš pracovný proces prebiehal postupne. Najskôr sme vytvorili špecifikáciu systému, ktorá definovala základné požiadavky a aktérov systému. Následne sme pripravili používateľské scenáre, ktoré nám pomohli lepšie pochopiť interakciu používateľov so systémom. Na základe týchto scenárov sme navrhli relačný model databázy a na záver sme spracovali zhodnotenie celého riešenia.

Pri tvorbe dokumentácie sme využili aj nástroje umelej inteligencie. Konkrétne sme ich použili na úpravu textu dokumentácie, opravu gramatických chýb a zlepšenie formátovania a štylistiky textu. Výsledný text sme však vždy manuálne skontrolovali a upravili tak, aby presne zodpovedal nášmu riešeniu.

Rámcovú tému e-shopu sme si zvolili preto, že naše pôvodné návrhy tém už boli obsadené. E-shop však predstavuje rozsiahlu a flexibilnú tému, ktorá umožňuje modelovať rôzne procesy a entity. Zároveň poskytuje dostatočný priestor na návrh realistického dátového modelu.
